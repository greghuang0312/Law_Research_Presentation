## 新项目开发顺序判断表

| 项目特征 | 建议顺序 | 适用信号 | 原因 |
|---|---|---|---|
| 交互和页面流程最不确定 | 先前端原型，再补后端 | 需求还在看页面、看流程、看表单交互；后端可先用 mock 数据替代 | 先把最容易返工的交互层定下来 |
| 业务规则、数据结构、权限、检索、生成链路最复杂 | 先后端能力，再接前端 | 真正难点在规则引擎、数据模型、合规、存储、接口编排 | 没有稳定后端，前端容易反复改接口和状态结构 |
| 接口边界已经清楚，前后端都能并行推进 | 前后端交叉开发 | 数据模型和 API 已经先定好；团队能接受 mock / stub 联调 | 总体效率最高，但前提是契约先稳定 |
| 内容型 / 营销型 / 展示型页面为主 | 前端优先 | 主要目标是视觉呈现、内容编排、转化路径 | 后端通常不是主风险 |
| 流程型 / 数据型 / 工具型系统为主 | 后端优先 | 重点在查询、生成、校验、任务流转、数据一致性 | 核心价值在能力链路而不是页面壳 |
| 多角色、多状态流转、复杂交互同时存在 | 先定接口契约，再交叉开发 | 页面和后端都复杂，但边界可先抽象清楚 | 避免两边各自演化，联调时大面积返工 |

## 默认判断规则

1. 先看“最大不确定性”在哪一层。
2. 如果最大不确定性在交互，先前端。
3. 如果最大不确定性在规则和数据，先后端。
4. 如果两边都复杂，就先定数据模型和 API，再交叉开发。
